# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A tiny reactive framework written as a teaching project for students learning JavaScript and the DOM, structured as an npm-workspaces monorepo. The framework lives in `packages/app.js` (TypeScript 7, strict, native `#private` internals; source of truth is two files, `packages/app.js/src/app.ts` for the DOM/reactivity layer and `packages/app.js/src/expression.ts` for the expression language); its build output `dist/` is **generated and gitignored — never commit build output**. A `prepare` script rebuilds `dist/` on every `npm install`. Runnable examples live in `packages/examples`, each served as its own web root by the zero-dependency `serve.mjs` (which aliases `/app.js` to the framework build). Framework runtime dependencies: none — keep it that way. Component files may carry a `<script>` (single-file components) — those mount as child Component instances with their own state, props (`data-component-prop-*`, one-way, batched `props` event), events (`data-component-on-*`, dedicated per-instance `EventTarget`), `mounted()` lifecycle, and `refs`; template-only files remain shared-scope includes.

## Commands

```sh
npm install         # dev deps + builds packages/app.js/dist via prepare
npm run build       # tsc → packages/app.js/dist
npm run typecheck   # all workspaces
npm test            # framework unit suite + examples smoke suite
npm run ex:counter  # serve the counter example on :8123
npm run ex:form     # serve the form example on :8123
npm run ex:todo     # serve the todo example on :8123
npm test -w app.js -- tests/components.test.ts   # single test file
```

Framework tests import `../src/app` directly; examples smoke tests drive the built `dist/` over real HTTP via `serve.mjs` + happy-dom's `Browser` (with `enableJavaScriptEvaluation: true`). Convention for newly found bugs: encode each as an `it.fails` case asserting the *desired* behavior (with its issue number in the test name) — once the bug is fixed, that test starts failing; remove the `.fails` modifier as part of the fix. No such markers are currently open.

Known upstream issue: happy-dom ≤ 20.10.6 silently drops falsy non-string `textContent` assignments (`el.textContent = 0` renders empty; browsers render `"0"`) — [capricorn86/happy-dom#2236](https://github.com/capricorn86/happy-dom/issues/2236), fix submitted as [PR #2237](https://github.com/capricorn86/happy-dom/pull/2237). The counter example sidesteps it via `${}` interpolation (always assigns `String(value)` to a Text node), so its smoke test asserts the zero render directly; the bug still affects `data-value` bindings that assign falsy non-strings — when a fixed happy-dom ships, bump the pin.

## Architecture

Everything is the `Component` class in `packages/app.js/src/app.ts`. One instance = one component tree rooted at `element` (default `document.body`). Internals are native `#private`; the public surface is the constructor, `element`, `data`, `methods`, `componentName`, `events`, `props`, `refs`, `ready` (a promise that settles when the initial mount finishes — rejections carry the original error, with a built-in `console.error` fallback; per-item components inside `data-for` blocks are never awaited by `ready`, initial pass included — their failures surface via their own fallback), `destroy()` (aborts every listener via a single `AbortController` signal, clears the binding maps and list registries, gates further update passes, and rejects a still-pending `ready` quietly — rendered DOM is left in place), `static loadTemplate`, and `static clearTemplateCache` (clears the template AND definition caches).

**Reactivity — `createGhost(data)`.** The constructor wraps `data` in a "ghost" object: each primitive key becomes a getter/setter pair over the original data, each object key recurses into a nested ghost; arrays are leaf values — replace them to update (`data.todos = [...data.todos, x]`; `push` alone doesn't trigger, and `data.todos = data.todos` is the sanctioned escape hatch after in-place mutation). Object keys now support the same self-assignment escape hatch as arrays (`data.user = data.user` re-renders after in-place mutation; wholesale replacement still throws, now loudly). Every set triggers `runUpdatePass()` — list reconciliation (`updateLists`), then `updateVisibility()`, then `updateValues()` — there is no dependency tracking; all bindings (and all list blocks) re-evaluate on any change. Ghosts are non-extensible, so the data shape is fixed at construction. Two-way input write-back assigns through the expression's dot path so the final ghost setter fires; the triggered pass skips re-writing the input being edited (the write-back source is consumed once at pass entry, so nested passes re-write normally).

**Templates and components.** `loadComponent()` fetches `/templates/<componentName>.html`, renders it, and appends the result into the wrapper element. Template fetches are cached in a static `Map` (template-name → promise) shared by all `Component` instances; a failed load — network error or non-`ok` HTTP response — rejects with the original error and is evicted from the cache so the next load retries. Each template file must have a `<template>` element as its first child — `renderTemplate` reads `divElement.firstChild.content`. Elements with `data-component="name"` inside a template recursively load that template as a sub-component; each recursion branch carries its own copy of the ancestor chain, so reuse across branches is fine while true cycles are rejected (issue #1). Template-only sub-components share the root instance's `data`/`methods` — there is no per-component scope (single-file components, above, are the exception: they get their own `data`/`methods`/`props`/`events`/`refs`).

**Directives** (wired up once, in `renderTemplate`):
- `data-show-if="expr"` — element is swapped with an anchor comment node when the expression is falsy (`hideElement`/`showElement`), not CSS-hidden.
- `data-display-if="expr"` — toggles inline `style.display` (`none` ↔ the element's original inline value) while the element stays in the DOM — the CSS-friendly counterpart to `data-show-if` (stable siblings/`:nth-child`; stylesheet `display` applies when the original inline value is empty). Allowed on a `data-for` element itself (no anchor conflict) and inside items, with item scope.
- `data-value="expr"` — two-way `value` binding for form controls only: `<input>`/`<textarea>` (via the `input` event) and `<select>` (via `change`). Checkbox/radio inputs are a loud error (their state is `checked`, not `value`), as is any non-form element — display text with `${}` interpolation instead. The expression must also be a plain dot path (an identifier, optionally followed by `.member` steps) — `[]` and `?.` steps aren't assignable, so a computed or optional expression there is a loud wiring-time error too, and a path rooted in a component prop is rejected the same way (props are inputs; copy into `data` to edit).
- `data-on-click` / `data-on-submit` — attribute value is a key in `methods`. Only these two events are supported; to add one, extend the module-level `eventNameList` in `app.ts`.
- `data-for="expr"` + required `data-key="expr"` — keyed list rendering. The element is the per-item template (replaced by an anchor-comment pair); clones are reconciled by `String(key)`: reuse/move/remove, first duplicate wins. Runtime list errors (throwing/non-array list expressions, throwing keys, duplicate keys) log once while they persist and re-arm after a clean pass. Item expressions see `$item`, `$index` (source-array index), `$array` (the list evaluated this pass). Not combinable with `data-show-if`/`data-component` on the same element; no nested `data-for` in the block. A `data-component` include inside items works only as a full single-file component (item-scoped props/events, one child instance per entry, re-seeded/destroyed as entries change) — a template-only include there is a loud, once-per-entry error. No form-control `data-value` inside items — all loud setup errors. Handlers inside items are invoked as `method(event, item, index)`, resolved at event time.
- `data-component-prop-<name>="expr"` / `data-component-on-<event>="method"` — component inputs and outputs (SFC wrappers only; loud errors on includes).

**Text interpolation.** Template text supports `${expression}` (escape a literal with `\${`); each placeholder becomes its own `Text` node binding — wired after `data-for` extraction (item scope applies inside blocks), re-evaluated every values pass, `null`/`undefined` render as empty string. A brace-counting scanner (not a regex) splits the text; an unmatched `${` is a loud wiring error and the text is left as written. Not supported inside `data-value` elements (`data-value` overwrites its content — loud error).

**Expression evaluation — `evaluate()`.** Directive and interpolation expressions run through a small hand-written pipeline in `packages/app.js/src/expression.ts` — a tokenizer, a Pratt parser, and a tree-walking evaluator — with no `eval`, no `Function`, and no other code-generation API involved. `compile(source)` parses once and caches the frozen, immutable `CompiledExpression` by its source string, so re-wiring the same expression text (another list item, another update pass) reuses the cached AST instead of reparsing. Identifier lookups run one fixed resolution chain — `$-scope → props → data → methods → globals whitelist` (`Math`, `JSON`, `Number`, `String`, `Boolean`, `Array`, `isNaN`, `isFinite`, `parseInt`, `parseFloat`) — where `$-scope` is the `$item`/`$index`/`$array` bindings inside a `data-for` item; a name that isn't found anywhere in the chain is a loud error. Pipes (`|>`) follow F#: `left |> right` calls `right(left)`, chaining left to right, and a bare (unparenthesized) ternary or arrow function on either side of `|>` is a parse error asking for explicit parentheses rather than a guessed precedence. Two-way write-back (`data-value`) only accepts a static dot path — an identifier or a chain of plain `.member` steps, never a computed `[]` or optional `?.` step — and writes through the ghost object's own setters so reactivity fires normally; the source element is recorded for the write and consumed at the very start of the next update pass, so that one binding is skipped when values are written back into form controls (a nested, re-entrant pass sees the slot already cleared and updates that control like any other). Sandboxing: only the whitelisted globals above are reachable, `constructor`/`__proto__`/`prototype` are blocked keys on every member access and call, and things like `in`/`instanceof`, bitwise operators, `**`, loose equality, and optional calls are rejected outright as not part of the language rather than silently misparsed. A malformed expression throws `ExpressionParseError` (carrying the source and the failing character's position) at wiring time — when a template mounts or a list item is created, not when it renders — and is logged once with a caret rendered under the failing character; the directive or block that owns it is dropped rather than left half-wired.

**Methods.** Bound to the component instance at construction, so `this` inside a method is the `Component` (giving access to `this.data`); they are invoked as `method(event, item, index)` — `item`/`index` carry values for handlers inside a `data-for` block (resolved at event time) and are `undefined` elsewhere.
