# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A tiny reactive framework written as a teaching project for students learning JavaScript and the DOM, structured as an npm-workspaces monorepo. The framework lives in `packages/app.js` (TypeScript 7, strict, native `#private` internals; source of truth is two files, `packages/app.js/src/app.ts` for the DOM/reactivity layer and `packages/app.js/src/expression.ts` for the expression language); its build output `dist/` is **generated and gitignored ‚Äî never commit build output**. A `prepare` script rebuilds `dist/` on every `npm install`. Runnable examples live in `packages/examples` (`counter/`, `form/`, `todo/`, `registration/`), each served as its own web root by the zero-dependency `serve.mjs` (which aliases `/app.js` to the framework build). Framework runtime dependencies: none ‚Äî keep it that way; the examples workspace is the one exception, carrying `zod` as a devDependency for the `registration/` capstone (`serve.mjs` separately aliases `/zod.js` to the installed package's own single-file ESM bundle ‚Äî `packages/app.js/package.json` itself stays untouched). Component files may carry a `<script>` (single-file components) ‚Äî those mount as child Component instances with their own state, props (`data-component-prop-*`, one-way, batched `props` event), events (`data-component-on-*`, dedicated per-instance `EventTarget`), `mounted()` lifecycle, and `refs`; template-only files remain shared-scope includes.

## Commands

```sh
npm install         # dev deps + builds packages/app.js/dist via prepare
npm run build       # tsc ‚Üí packages/app.js/dist
npm run typecheck   # all workspaces
npm test            # framework unit suite + examples smoke suite
npm run ex:counter      # serve the counter example on :8123
npm run ex:form         # serve the form example on :8123
npm run ex:todo         # serve the todo example on :8123
npm run ex:registration # serve the registration capstone example on :8123
npm test -w app.js -- tests/components.test.ts   # single test file
```

Framework tests import `../src/app` directly; examples smoke tests drive the built `dist/` over real HTTP via `serve.mjs` + happy-dom's `Browser` (with `enableJavaScriptEvaluation: true`). Convention for newly found bugs: encode each as an `it.fails` case asserting the *desired* behavior (with its issue number in the test name) ‚Äî once the bug is fixed, that test starts failing; remove the `.fails` modifier as part of the fix. No such markers are currently open.

Known upstream issue: happy-dom ‚â§ 20.10.6 silently drops falsy non-string `textContent` assignments (`el.textContent = 0` renders empty; browsers render `"0"`) ‚Äî [capricorn86/happy-dom#2236](https://github.com/capricorn86/happy-dom/issues/2236), fix submitted as [PR #2237](https://github.com/capricorn86/happy-dom/pull/2237). The framework no longer has any path that can hit it: `${}` interpolation always assigns `String(value)` to a Text node (which is also why the counter smoke asserts the zero render directly), and `data-value` writes `.value`/`.checked`, never `textContent`. The note remains only to explain the interpolation String() choice and the exact pin ‚Äî bump it when a fixed happy-dom ships.

## Architecture

Everything is the `Component` class in `packages/app.js/src/app.ts`. One instance = one component tree rooted at `element` (default `document.body`). Internals are native `#private`; the public surface is the constructor, `element`, `data`, `methods`, `componentName`, `events`, `props`, `refs`, `ready` (a promise that settles when the initial mount finishes ‚Äî rejections carry the original error, with a built-in `console.error` fallback; per-item components inside `data-for` blocks are never awaited by `ready`, initial pass included ‚Äî their failures surface via their own fallback), `updated()` (resolves once the flush pending at call time finishes draining; resolves immediately when idle, and after `destroy()` too), `destroy()` (aborts every listener via a single `AbortController` signal, clears the binding maps, list registries, and dependency graph, gates further flushes, resolves a still-pending `updated()` promise, and rejects a still-pending `ready` quietly ‚Äî rendered DOM is left in place), `static loadTemplate`, and `static clearTemplateCache` (clears the template AND definition caches).

**Reactivity ‚Äî `createGhost(data)`.** The constructor wraps `data` in a "ghost" object: each key's getter/setter is stamped with its dot path from the root (`user.address.city`); a primitive or array key reads/writes a leaf value directly, a plain-object key recurses into its own nested ghost carrying that longer path. Arrays stay leaf values ‚Äî replace them to update (`data.todos = [...data.todos, x]`; `push` alone doesn't trigger). Reading a ghost path only matters while a tracking frame is open: every expression evaluation ‚Äî a directive, an interpolation, a `data-for` list expression, a component prop ‚Äî opens a fresh frame, and whichever paths its getters touch before it returns (even by throwing) become that binding's subscriptions, replacing whatever it read the previous time (a `flag ? a : b` binding drops the untaken branch's path and picks up the taken one the moment `flag` flips). A binding whose frame comes back empty ‚Äî a pure literal, or an expression that throws before reading anything ‚Äî never subscribes to anything, so no later write can notify it into re-running; outside a `data-for` item (where reconciling the block re-dirties every entry's bindings regardless, see below), that means it renders ‚Äî or logs its error ‚Äî exactly once, for good. A write passes through a gate first: equal primitives and double-`null` are a no-op ‚Äî nothing notifies ‚Äî but an equal object, array, or function reference still goes through unsuppressed; that asymmetry is the deliberate escape hatch for in-place mutation (mutate `data.todos`'s contents, then `data.todos = data.todos`; same for `data.user = data.user`, or a mid-chain `data.user.address = data.user.address`) ‚Äî a plain object otherwise stays replace-only, throwing loudly on any other value. A write that clears the gate notifies every binding subscribed to that exact path AND to any path registered as its descendant (prefixed `path.`) ‚Äî never an ancestor, since nothing above the written path changed identity. Notified bindings land in a dirty set rather than re-rendering there and then: the first write in a tick schedules a microtask flush and mints that flush's `updated()` promise at schedule time (per instance), so every write landing in the same tick ‚Äî however many ‚Äî collapses into that one promise. The flush drains the dirty set in ordered passes ‚Äî list blocks, `data-show-if`, `data-display-if`, `data-disabled-if`, `data-value`, text, then component prop re-seeds ‚Äî looping while a pass leaves fresh dirty bindings behind (a formatter that both reads and writes its own path re-dirties itself every drain) and giving up with a loud console error after 64 iterations rather than hanging. A `data-for` item's `$item`/`$index`/`$array` bindings are never path-tracked, so reconciling a block ‚Äî itself only woken by a write to its own list path, including the array self-assign hatch ‚Äî unconditionally marks every surviving entry's bindings dirty regardless of whether anything about the entry looks different; that unconditional marking is the only channel through which an in-place item mutation, surfaced via the hatch, reaches the DOM. No tracking frame is open at event-handler time or around a `data-key` expression, so reads there never subscribe anything ‚Äî a method peeking at `this.data.x`, or a key expression reading a value outside `$item`, has no bearing on what re-renders later. (An earlier, coarser engine ‚Äî no tracking, every write re-running every binding synchronously ‚Äî is preserved in this repository's history for comparison, not in the current source.) Ghosts are non-extensible, so the data shape is fixed at construction. Two-way input write-back assigns through the expression's dot path so the ghost's own setter fires; the drain then skips writing any `data-value` control whose current state already equals what it would write ‚Äî which is exactly the input being typed into, so its caret survives without any source bookkeeping (a derived write to the same path produces a different value and lands normally).

**Templates and components.** `loadComponent()` fetches `/templates/<componentName>.html`, renders it, and appends the result into the wrapper element. Template fetches are cached in a static `Map` (template-name ‚Üí promise) shared by all `Component` instances; a failed load ‚Äî network error or non-`ok` HTTP response ‚Äî rejects with the original error and is evicted from the cache so the next load retries. Each template file must have a `<template>` element as its first child ‚Äî `renderTemplate` reads `divElement.firstChild.content`. Elements with `data-component="name"` inside a template recursively load that template as a sub-component; each recursion branch carries its own copy of the ancestor chain, so reuse across branches is fine while true cycles are rejected (issue #1). Template-only sub-components share the root instance's `data`/`methods` ‚Äî there is no per-component scope (single-file components, above, are the exception: they get their own `data`/`methods`/`props`/`events`/`refs`).

**Directives** (wired up once, in `renderTemplate`):
- `data-show-if="expr"` ‚Äî element is swapped with an anchor comment node when the expression is falsy (`hideElement`/`showElement`), not CSS-hidden.
- `data-display-if="expr"` ‚Äî toggles inline `style.display` (`none` ‚Üî the element's original inline value) while the element stays in the DOM ‚Äî the CSS-friendly counterpart to `data-show-if` (stable siblings/`:nth-child`; stylesheet `display` applies when the original inline value is empty). Allowed on a `data-for` element itself (no anchor conflict) and inside items, with item scope.
- `data-disabled-if="expr"` ‚Äî truthy toggles `element.disabled = true`, falsy `false`. Mirrors `data-display-if` structurally: its own tracked-binding kind, draining alongside display, wired at both the root sweep and in-item sites (item scope), allowed on a `data-for` element itself. Allowed only on elements that honor `disabled` ‚Äî `input`, `textarea`, `select`, `button` ‚Äî a loud setup error names the rule anywhere else. Composes freely with `data-value` on the same control (a disabled control still renders programmatic writes; the browser suppresses user events while disabled, so write-back simply never fires ‚Äî no framework special case) and with `data-show-if`/`data-display-if` (independent bindings, each dirtied and evicted on its own).
- `data-value="expr"` ‚Äî two-way binding for form controls only. Text `<input>`/`<textarea>` bind `.value` via the `input` event; `<select>` binds `.value` via `change`. Checkbox inputs bind a boolean through `.checked` (`change` event): render is `el.checked = !!value`, write-back stores the real boolean `el.checked`. Radio inputs get per-element group semantics ‚Äî every radio in the group carries its own binding evaluating the *same* expression, renders `el.checked = (value === el.value)`, and writes back `el.value` on `change`; only the shared expression ties the group together, there is no separate group registration. File inputs remain a loud error (a browser won't let script set `.files` ‚Äî handle their `change` event with `data-on-change` instead), as is any non-form element ‚Äî display text with `${}` interpolation instead. The expression must also be a plain dot path (an identifier, optionally followed by `.member` steps) ‚Äî `[]` and `?.` steps aren't assignable, so a computed or optional expression there is a loud wiring-time error too, and a path rooted in a component prop is rejected the same way (props are inputs; copy into `data` to edit).
- `data-on-<event>` ‚Äî binds any DOM event by name: the attribute's kebab suffix is used verbatim as the event type (`data-on-change`, `data-on-keydown`, `data-on-pointerdown`, custom events ‚Äî all work, no enumeration needed), attribute value is a key in `methods`. HTML lowercases attribute names, so case-sensitive event types are inexpressible ‚Äî irrelevant for the all-lowercase DOM event names this covers. A typo'd event name binds and waits silently forever ‚Äî there is no unknown-name check.
- `data-for="expr"` + required `data-key="expr"` ‚Äî keyed list rendering. The element is the per-item template (replaced by an anchor-comment pair); clones are reconciled by `String(key)`: reuse/move/remove, first duplicate wins. Runtime list errors (throwing/non-array list expressions, throwing keys, duplicate keys) log once while they persist and re-arm after a clean pass. Item expressions see `$item`, `$index` (source-array index), `$array` (the list evaluated this pass). Not combinable with `data-show-if`/`data-component` on the same element; no nested `data-for` in the block. A `data-component` include inside items works only as a full single-file component (item-scoped props/events, one child instance per entry, re-seeded/destroyed as entries change) ‚Äî a template-only include there is a loud, once-per-entry error. No form-control `data-value` inside items ‚Äî all loud setup errors. Handlers inside items are invoked as `method(event, item, index)`, resolved at event time.
- `data-component-prop-<name>="expr"` / `data-component-on-<event>="method"` ‚Äî component inputs and outputs (SFC wrappers only; loud errors on includes).

**Text interpolation.** Template text supports `${expression}` (escape a literal with `\${`); each placeholder becomes its own `Text` node binding ‚Äî wired after `data-for` extraction (item scope applies inside blocks), tracked like any other binding (re-evaluated only when a path it reads is written; once and never again if it reads nothing reactive), `null`/`undefined` render as empty string. A brace-counting scanner (not a regex) splits the text; an unmatched `${` is a loud wiring error and the text is left as written. Not supported inside `data-value` elements (`data-value` overwrites its content ‚Äî loud error).

**Expression evaluation ‚Äî `evaluate()`.** Directive and interpolation expressions run through a small hand-written pipeline in `packages/app.js/src/expression.ts` ‚Äî a tokenizer, a Pratt parser, and a tree-walking evaluator ‚Äî with no `eval`, no `Function`, and no other code-generation API involved. `compile(source)` parses once and caches the frozen, immutable `CompiledExpression` by its source string, so re-wiring the same expression text (another list item, another drain) reuses the cached AST instead of reparsing. Identifier lookups run one fixed resolution chain ‚Äî `$-scope ‚Üí props ‚Üí data ‚Üí methods ‚Üí globals whitelist` (`Math`, `JSON`, `Number`, `String`, `Boolean`, `Array`, `isNaN`, `isFinite`, `parseInt`, `parseFloat`) ‚Äî where `$-scope` is the `$item`/`$index`/`$array` bindings inside a `data-for` item; a name that isn't found anywhere in the chain is a loud error. Pipes (`|>`) follow F#: `left |> right` calls `right(left)`, chaining left to right, and a bare (unparenthesized) ternary or arrow function on either side of `|>` is a parse error asking for explicit parentheses rather than a guessed precedence. Two-way write-back (`data-value`) only accepts a static dot path ‚Äî an identifier or a chain of plain `.member` steps, never a computed `[]` or optional `?.` step ‚Äî and writes through the ghost object's own setters so reactivity fires normally; the drain skips the DOM write when the control's current state already equals what it would write ‚Äî true by definition for the input being typed into, which keeps its caret; a derived write produces a different value and lands normally. Sandboxing: only the whitelisted globals above are reachable, `constructor`/`__proto__`/`prototype` are blocked keys on every member access and call, and things like `in`/`instanceof`, bitwise operators, `**`, loose equality, and optional calls are rejected outright as not part of the language rather than silently misparsed. A malformed expression throws `ExpressionParseError` (carrying the source and the failing character's position) at wiring time ‚Äî when a template mounts or a list item is created, not when it renders ‚Äî and is logged once with a caret rendered under the failing character; the directive or block that owns it is dropped rather than left half-wired.

**Methods.** Bound to the component instance at construction, so `this` inside a method is the `Component` (giving access to `this.data`); they are invoked as `method(event, item, index)` ‚Äî `item`/`index` carry values for handlers inside a `data-for` block (resolved at event time) and are `undefined` elsewhere.
